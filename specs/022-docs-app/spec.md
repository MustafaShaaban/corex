# Feature Specification: Documentation web app

**Feature Branch**: `022-docs-app`

**Created**: 2026-06-11

**Status**: Draft (RETROSPECTIVE — documents delivered, build-verified code; item 8 of the "Finish Corex" initiative; reconciled to the implementation in `docs-app/`)

**Input**: "A real documentation website for Corex — getting started, guides, architecture, and a searchable per-class internals reference — that describes the actual code and builds to a static site."

> **Retrospective note.** Written after the site shipped, to restore spec-first compliance (Principle X). The
> hand-written pages and the site shell are this feature (item 8); the generated per-class `reference/` pages
> are produced by `wp corex docs:generate` (spec 019, item 9) and consumed here. Requirements describe the
> existing `docs-app/` (Astro + Starlight) project.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Learn and use Corex from a real docs site (Priority: P1)

A developer opens the Corex docs site and finds an introduction, a getting-started path (WAMP/Apache+WP-CLI,
wp-env/Docker, monorepo wiring, first run + brand), task guides (forms, blocks via CLI, queries, branding, the
CLI, configuration, mail, MVC), an architecture overview, an internals reference, an FAQ, and troubleshooting —
all describing the real code, with working search and navigation.

**Why this priority**: A framework without usable docs is unusable by anyone but its author. This is the
on-ramp.

**Independent Test**: Run `npm run build` in `docs-app/`; confirm it builds a static site with the authored
pages + the reference pages, a client-side search index, and the navigation (sidebar, breadcrumbs, prev/next).

**Acceptance Scenarios**:

1. **Given** the `docs-app/` project, **When** `npm run build` runs, **Then** it produces a static `dist/`
   site including every authored page and the generated reference pages, plus a Pagefind search index.
2. **Given** the running site, **When** a reader searches, **Then** Pagefind returns matches client-side (no
   server), and the left-nav / breadcrumbs / prev-next / light-dark / copy-buttons all work.
3. **Given** an authored page (e.g. the Mail guide), **When** read, **Then** its API examples match the real
   code (the Mail guide was corrected to `template()->with()` against `MessageBuilder`).
4. **Given** the RTL requirement, **When** an Arabic locale is added, **Then** Starlight renders it
   right-to-left (the shell is RTL-ready; English ships now, AR is additive).

### Edge Cases

- `node_modules/` and `dist/` are git-ignored; the site is built from source on checkout/CI.
- The generated `reference/*/` pages are git-ignored (regenerated by `docs:generate`); the hand-written
  `reference/index.md` is kept.
- The site builds with `base: '/'` so it can move to a dedicated public site unchanged; a repo-subpath build
  is a documented one-line config change.
- A missing `site` value yields only a cosmetic sitemap warning (no public canonical URL yet).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The docs app MUST be a static-site project (Astro + Starlight) under `docs-app/`, building to
  `dist/` with `npm run build` and served locally with `npm run dev`.
- **FR-002**: It MUST author the core documentation set: Introduction; Getting Started (overview, WAMP+WP-CLI,
  wp-env/Docker, monorepo wiring, first run + brand); Guides (forms, blocks-via-CLI, queries, branding, CLI,
  configuration, mail, MVC); Architecture overview; Internals Reference index; FAQ; Troubleshooting.
- **FR-003**: Every authored page MUST describe the **real** code (no invented APIs); API examples MUST match
  the source.
- **FR-004**: The site MUST provide client-side search (Pagefind), left-nav, breadcrumbs, prev/next, light/dark,
  copy buttons, and be RTL-ready (an Arabic locale renders right-to-left).
- **FR-005**: It MUST consume the generated per-class `reference/` pages (spec 019) into the reference
  navigation, while keeping the hand-written reference index.
- **FR-006**: `node_modules/`, `dist/`, and the generated `reference/*/` pages MUST be git-ignored; the site
  MUST build from source (a documented checkout/CI step).

### Key Entities

- **Docs site**: the Astro + Starlight project (`astro.config.mjs`, content collection, sidebar).
- **Authored page**: a hand-written Markdown/MDX page under `src/content/docs/`.
- **Reference page**: a generated per-class page (spec 019) surfaced in the reference nav.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `npm run build` completes green and emits a static `dist/` with the authored pages + the
  reference pages + a search index.
- **SC-002**: The authored core set (getting-started, guides, architecture, FAQ, troubleshooting) is present
  and navigable.
- **SC-003**: Client-side search returns results with no server.
- **SC-004**: Documented APIs match the source (the Mail guide verified against `MessageBuilder`).
- **SC-005**: A fresh checkout builds the site after `npm install` (no committed `node_modules`/`dist`).

## Assumptions

- The generated `reference/` pages come from `wp corex docs:generate` (spec 019); this feature owns the site
  shell + the hand-written pages.
- The build is a Node step (this headless WAMP runs it; the rendered site's visual polish needs a browser).
- A future public deployment sets `site`/`base` and adds the AR locale content.
